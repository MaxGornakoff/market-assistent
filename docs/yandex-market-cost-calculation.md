# Как рассчитывается показатель «Расходы Маркета»

## Назначение показателя

Поле «Расходы Маркета» показывает оценку расходов по конкретному товару на основе данных, которые возвращает API Яндекс Маркета. Это не вручную введённое значение и не статическая цифра. Показатель рассчитывается динамически при загрузке списка отслеживаемых товаров.

Итоговая сумма формируется как сумма тарифных составляющих, которые Маркет возвращает для товара: комиссия, перевод платежа, доставка и другие сервисные расходы.

---

## Общая схема расчёта

1. Фронтенд открывает раздел управления ценами.
2. Компонент отправляет запрос в наш backend:
   - GET /api/yandex-market/products
3. Backend получает список локально отслеживаемых товаров из таблицы yandex_market_products.
4. Для каждого offer_id backend запрашивает у Яндекс Маркета актуальные данные по товару.
5. На основе полученной цены, категории и габаритов backend делает отдельный расчёт тарифов через API Маркета.
6. Все тарифные компоненты суммируются.
7. Готовое значение возвращается во фронтенд в поле market_service_cost.

---

## Какие API-методы используются

### 1. GET /v2/campaigns

Используется для:
- проверки подключения к кабинету Маркета
- определения businessId
- получения списка активных campaignId

Это вспомогательный метод. Он нужен, чтобы понять, с каким бизнесом и какими кампаниями работать дальше.

### 2. POST /v2/businesses/{businessId}/offer-mappings

Это основной источник данных по самим товарам.

Из ответа мы берём:
- offerId
- name
- basicPrice.value
- basicPrice.currencyId
- basicPrice.updatedAt
- mapping.marketCategoryId
- mapping.marketSku
- showcaseUrl
- weightDimensions

Именно отсюда получаем актуальную цену товара и данные, нужные для последующего расчёта расходов.

### 3. POST /v2/tariffs/calculate

Этот метод используется непосредственно для расчёта «Расходов Маркета».

В него отправляется набор параметров по товару:
- categoryId
- price
- length
- width
- height
- weight

Дополнительно передаются параметры расчёта:
- либо campaignId, если у нас есть активная кампания
- либо sellingProgram и currency, если campaignId недоступен

Важно: API Маркета строго проверяет совместимость параметров. Например, нельзя одновременно передавать campaignId вместе с некоторыми другими конфликтующими полями. Эта логика в проекте уже учтена.

## Логика расчёта итоговой суммы

После вызова метода расчёта тарифов Маркет возвращает список элементов tariffs.

Примеры типов расходов:
- FEE — размещение или комиссия Маркета
- AGENCY_COMMISSION — агентская комиссия
- PAYMENT_TRANSFER — перевод платежа
- DELIVERY_TO_CUSTOMER — доставка покупателю
- CROSSREGIONAL_DELIVERY — межрегиональная доставка
- EXPRESS_DELIVERY — экспресс-доставка
- SORTING — обработка заказа
- MIDDLE_MILE — логистика между этапами доставки

Итоговый показатель считается так:

Расходы Маркета = сумма всех значений tariffs.amount по товару

То есть мы не берём только одну комиссию, а суммируем все составляющие, которые вернул API для конкретной позиции.

---

## Что происходит, если габариты товара отсутствуют или некорректны

У части товаров Маркет может вернуть пустые или нулевые значения длины, ширины, высоты или веса.

Если отправить такие значения в расчёт тарифов, API может вернуть ошибку вида:
- height must be greater than 0
- width must be greater than 0
- weight must be greater than 0

Чтобы расчёт не ломался, в проекте применяется безопасный fallback:
- length = 10
- width = 10
- height = 10
- weight = 0.5

Если использовались такие базовые значения, во фронтенд также передаётся пояснение, что оценка была рассчитана по API Маркета с базовыми габаритами товара.

Это позволяет:
- не терять расчёт полностью
- показывать пользователю ориентировочную стоимость
- избегать пустого столбца при проблемах с исходными габаритами

## Какие поля backend отдаёт во фронтенд

Для каждой строки таблицы backend формирует и возвращает следующие поля:
- market_price
- market_price_currency
- market_price_updated_at
- market_service_cost
- market_service_cost_currency
- market_service_cost_breakdown
- market_service_cost_note
- market_category_id
- market_sku
- market_url

Фронтенд уже использует эти поля для:
- отображения актуальной цены
- отображения расходов Маркета
- показа детализации в карточке товара

## Где в коде реализована логика

### Backend

Основная бизнес-логика находится в следующих местах:
- backend/app/Integrations/YandexMarket/YandexMarketClient.php
- backend/app/Http/Controllers/Api/YandexMarketProductController.php

Что делает каждая часть:
- YandexMarketClient.php — общается с API Маркета, собирает данные по товарам и считает тарифы
- YandexMarketProductController.php — формирует API-ответ для фронтенда и добавляет к локальным товарам рассчитанные метрики

### Frontend

Отображение результата находится здесь:
- frontend/src/components/dashboard/sections/yandex-market/PriceManagementPanel.vue

Этот компонент:
- загружает список товаров
- показывает столбец «Расходы Маркета»
- выводит детальную разбивку расходов в карточке товара

## Важные ограничения и особенности

1. Значение является оценочным
   - это расчёт по API Маркета на текущий момент
   - итоговые реальные удержания могут немного отличаться в зависимости от логистики, программы продаж и изменений тарифов

2. Значение зависит от полноты данных
   - если нет цены или категории, рассчитать расходы нельзя
   - если нет корректных габаритов, используется безопасная базовая оценка

3. Значение обновляется при загрузке таблицы
   - при каждом обновлении списка backend заново подтягивает данные из Маркета

4. На результат влияет активная кампания
   - если у кабинета несколько campaignId, расчёт опирается на активную кампанию, доступную в настройках

---

## Краткое резюме

Показатель «Расходы Маркета» в проекте считается не вручную, а автоматически через API Яндекс Маркета.

Источник расчёта:
- актуальная карточка товара из offer-mappings
- тарифный расчёт из tariffs/calculate

Формула итогового значения:
- сумма всех тарифных составляющих, которые возвращает Маркет по этому товару

Если исходные данные неполные, система использует безопасные базовые параметры, чтобы пользователь всё равно видел оценку, а не пустое поле.
